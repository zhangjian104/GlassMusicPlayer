# 现状分析：音乐播放器向有声小说平台的转型

本文档分析了 `GlassMusicPlayer` 现有的音乐网站结构与目标有声小说（基于 `desc.json` 数据结构）之间的差异和改造难点，旨在为后续的开发与重构提供指导方向。

## 1. 核心数据模型的差异

| 维度 | 现有音乐网站 (Music) | 目标有声小说 (Audiobook) | 结构错位点 |
| :--- | :--- | :--- | :--- |
| **基础单元** | 单曲 (Song) | 章节 (Episode) | 音乐是独立的，可随意组合；小说章节**必须依附于某本书**，且有严格的**先后顺序**。 |
| **集合方式** | 歌单 (Playlist) / 专辑 (Album) | 书 (Book) | 音乐平台中，一首歌可以同时存在于几十个歌单中；而小说中，章节是专属该书的。目前前端是用“歌单(Playlist)”的页面来强行承载“书(Book)”的概念。 |
| **创作者** | 歌手 (Artist) | 演播者 / 原著作者 (Author/Anchor) | 结构基本匹配，但演播者页面（原歌手页）应该展示的是**TA播讲的书（合集）**，而不是**TA最火的单曲（章节）**。没人会去听演播者随机的“热门第五集”。 |

## 2. 现状分析与存在的“违和感”

目前项目通过国际化替换了表面词汇（如：歌曲->章节，歌单->书单），并修改了首页的“热门主播”数据源，掩盖了表面的差异，但在深层逻辑上依然存在明显的“音乐网站残留”：

### 🔴 A. 播放逻辑的不合理
* **随机播放 (Shuffle)**：例如在书单详情页 (`playlist.vue`) 中依然有**“随机播放”**按钮。对于有声小说来说，随机播放章节是反直觉、反业务逻辑的。
* **播放进度记忆**：听歌没听完，下次通常从头听。但有声小说一集可能长达几十分钟，用户关掉网页后，需要**精准记忆听到哪一本书的哪一集的哪一秒**。目前的音乐播放器通常只记忆“上次听的歌”，缺乏精准的进度同步能力。

### 🔴 B. 演播者（原歌手）页面的结构错位
* 在当前逻辑下，点击某个演播者（如“青雪”）进入详情页，依然会去请求并展示“该主播的热门单曲 50 首”。
* 在有声小说的逻辑下，该页面应该直接展示 `desc.json` 中属于该演播者的**书籍合集**（如《盗墓笔记第二季》），以书籍卡片列表的形式呈现，而不是直接平铺 MP3 章节。

### 🔴 C. 首页数据的来源错位
* 目前首页除了“热门主播”接入了本地的聚合接口外，其他的模块（如：**轮播图、推荐书单、最新章节**）依然在通过 Vite proxy 请求网易云的 API 获取测试数据。
* 按照 `desc.json` 的结构，首页的“推荐书单”其实应该直接对应 `books` 数组里的书籍列表。

### 🔴 D. 图片与媒体资源的获取方式
* 原本的网易云 API 图片直接返回带 `http/https` 的全路径，且前端通过拼接 `?param=xxx` 裁剪。
* 目标系统的 `desc.json` 中，无论是 `anchor_avatars` 还是 `cover` 都是**相对路径**（如 `qing_xue/daomu_biji_season2/cover.png`）。我们需要在前端全局处理，将其拼接成指向后端的绝对地址，同时避免前端裁剪参数污染后端的图片请求（目前已通过 `URL#` 哈希锚点作为临时解决方案截断了参数）。

## 3. 下一步的改造建议 (BFF 层与前端适配)

为了让这个项目彻底蜕变为有声小说网站，后续需要针对 `desc.json` 的结构对前端进行以下针对性改造：

1. **废弃无用且违和的功能**：在前端路由和组件中彻底隐藏或删除“随机播放”、“单曲循环”、“喜欢/收藏单曲”等音乐专属操作（小说通常是列表顺序播放，且只收藏整本书）。
2. **重构演播者页 (`artist.vue`)**：将原来展示“热门歌曲”的列表区域，改造为展示“该主播的全部作品 (Books)”。
3. **播放状态精准持久化**：深度改造 Pinia 中的 `audio` store，在 `timeupdate` 事件时，不仅记录当前章节 ID，还要将**当前书籍 ID**和**当前播放秒数**持久化到 `localStorage` 中。实现用户下次进入时，自动跳到上次听书的精确进度。
4. **全面切断网易云 API**：彻底替换首页及其他页面的数据源，将“推荐书单”、“轮播图”等彻底改为拉取本地依据 `desc.json` 解析生成的音频资源接口。

**总结**：当前项目阶段属于**“披着小说皮的音乐播放器”**。其 UI 框架（响应式、毛玻璃质感）非常优秀且契合，但其底层的**“顺序播放为主”**和**“以书为绝对核心”**的业务逻辑，仍需对现有 Vue 组件和状态管理进行一轮深度的业务修剪与定制。
